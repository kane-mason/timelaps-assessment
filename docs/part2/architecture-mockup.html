Timelaps · Part 2 · Design mockup

AI Analyst — Architecture

Ask the data a question. Get a plain-language answer and one chart, every figure recomputed from the tracker via lib/metrics.mjs and married to its base size on screen. The honest "this sample can't answer that" is a designed behaviour, not a footnote.

ScopeDeliberately scoped to the provided dataset (two waves, ~250 respondents), per the brief's "same dataset" — not built for arbitrary scale (no database, no RAG). By design.
01

The loop

A bounded Plan → Compute → Critique loop — Critique loops back to Plan on a dead-end — on one principle: encode invariants, not procedures. The model plans and re-plans the tool calls and narrates the answer; Compute and Critique — the steps that produce and vet the numbers — are deterministic code. The hinge is that Critique is plain code, not a second LLM — a guard pass over a provenance-carrying compute log.

Who decides to exit — the model, or the guard? Both. Parse is a structured intake (no tools); then Plan → Compute → Critique is one agentic loop. The model drives it and decides when it's done by calling submit_answer (carrying its answer + vizSpec) instead of another tool. That submission goes through Critique — the gate, which validates it against the ledger: cleared → the answer flows to Render; vetoed (it leans on a barred cell, or a number not in the ledger) → the guard emits a code-authored finding — what's inadmissible, the boundary, the solid alternative — and the model goes again. So the LLM decides to stop; the guard decides whether the answer is allowed out. Bounded to 3 re-plans; on exhaustion the guard forces a deterministic honest fallback (state the reliability boundary), so the loop always ends.

LLM picks intent, drives tool-use & narrates compute deterministic guard the honesty hinge i/o what the leader sees
Parse LLM

Not a separate LLM call — Parse is the understanding half of the model's first turn (one call also does the first Plan). It commits to an intent (explain_movement, attribute_to_segment, locate_gap, size_opportunity, characterise_loyalty, describe_market, OUT_OF_SCOPE) + entity slots that can only hold members of the live vocabularies, and echoes the reading back ("Reading this as: Aurora, consideration, Q4 → Q1").

Plan LLM

Inside the loop: the model calls a registered tool — or, when it's done gathering, calls submit_answer (carrying its plain-language answer + a VizSpec) to finish. ↻ The loop returns here to call the next tool, or — on a veto — to form a new plan with the guard's finding. Honesty-critical steps (mix-adjust a movement, cross-check a thin cut) aren't left to the model — Critique appends them as invariants.

Compute compute

Each call dispatches to a lib/metrics.mjs export against responses_raw.json. Round-half-to-even lives inside the functions, so output byte-matches analysed_outputs.json (31.25→31.2). Results append to the ComputeLog with full provenance — in a fixed typed shape (below).

Critique — the gate guard

Plain code, no LLM — reads the typed result fields. THIN guard (base & count), NULL guard (rate===null → "not computable"), MIX guard, PROJECTION quarantine. It checks each result and validates the submit_answer against the ledger: cleared → the answer ships; veto (leans on a barred cell, or a number not in the ledger) → a code-authored finding (inadmissible cells + boundary + solid alternative) → back to Plan.

Render i/o

Deterministic, no model. Takes the cleared submission (answer + VizSpec), binds the VizSpec's refs → ledger values → draws the chart (themed, thin cells greyed), then lays out the card: headline → chart → Trust strip (base_n + confidence glyph + denominator lens) → "Show the working" → Caveat rail. Loud only when there's something to flag.

Trace i/o

Each figure expands to fn(args) → {value, count, base_n, low_confidence}. Footer: "recomputed from data; pinned in CI by /verify-data."

Why a deterministic guard is even possible — the typed registry

Critique can be plain code only because Compute always returns a fixed, typed shape. Every one of the 17 lib/metrics.mjs functions returns one of a small set of return types, each carrying its own base size and confidence flag — so the guard reads known fields off known objects, never free-form output. That consistency is the foundation the whole loop rests on.

Compute returns — always one of these

a delta/gap { value, … }
a what-if   { is_projection:true, assumption, … }

Pure functions + round-half-to-even → same input, same output → repeatable, and pinned in CI (/verify-data). Consistent shapes upstream are what make a deterministic guard — and the whole honesty story — possible.
02

The two guarantees

Grounded no invented numbers

Three structural locks, none of them prompt etiquette. The model decides which function and how to say it; the function decides the value; the loop decides whether that value may become a claim.

  • Constrained tool surface. The model's entire numeric vocabulary is the 17 functions in lib/metrics.mjs; fn is an enum, args type-checked. Handing the model the wheel on what to compute costs nothing here — who sequences the tools is irrelevant to whether a number can be invented.
  • Plan-then-compute. Raw rows never enter the model's context, so it cannot do mental math. Rounding lives inside the functions.
  • Claim-ledger post-check. Any prose number absent from a logged value fails the response. Derived figures (the +4.6pt gain) are first-class typed ledger entries from pure combinators — never narrator arithmetic.

CI anchor: /verify-data pins every function to analysed_outputs.json to the decimal; a golden-transcript test pins the full plan + ledger for Q1 and Q2. analysed_outputs.json is the CI fixture only — never the analyst's data source.

Thin data n < 50 is a loop stage

The brief's Q2 trap defeats a naive single check, so the guard runs two:

  • Check 1 — base: base_n < 50. Catches 50+ (n = 16 → 27).
  • Check 2 — numerator: count < 30, even when the band base clears 50. Catches the 25-34 "swing" (count 23 → 26 on base 88/74) and the 50+ used cell (count 4).

A unified causal_admissible flag (base_n ≥ 50 AND count ≥ 30) drives the UI confidence dot — so a cell the guard bars can never render a solid dot while the prose refuses it.

GATE→ FLAG (greyed, n + count)→ REDIRECT to a significant cut

The redirect is instrumented. Gender bases are solid (n = 98 / 136) but the female base shrank 124 → 98 and the −14 vs −5 contrast is within sampling margin — so gender is the directional read, not a confirmed driver.

03

Viz selection — the model picks the form, code owns the values

Same principle as the loop: open where the model reasons, code-owned where honesty lives. The model emits a VizSpec — mark / encoding / annotation — carrying only references into the ledger, never numbers. A deterministic renderer binds the values, applies the Timelaps theme, and greys thin cells. So the analyst can chart a question in a shape we never pre-designed — and a stronger model picks better forms for free — while it stays impossible to draw a number it didn't compute. A fixed dataset doesn't mean a fixed chart set.

Open — the model's job

Which chart, encoding, annotation — into an open, brand-themed, legibility-bounded grammar (Vega-Lite-style, or a bespoke ref-only schema). Improves with every model generation, no code change.

Code-owned — the invariants

The values (bound from the ledger, ref-only) and the confidence treatment (thin cells greyed). The model can't supply a number or un-grey a thin base. Openness of form ≠ openness of numbers.

Legibility is time-to-comprehend, not time-to-answer. An AI analyst is expected to think — the loop can take a beat and stream a "thinking" state. The bar (brief: "act on in seconds," "one glance") applies to the rendered card: clear, not fast.

Branded presets, not a cage — the model's default pick for common shapes (kept pixel-clean and on-brand); it composes a new form within the grammar for questions we didn't anticipate.
